## Summary

This report details the full static and dynamic analysis of a dual-behavior Android application, `com.circlelanes.trains`. The application, masquerading as a benign puzzle game, employs a sophisticated, multi-stage cloaking mechanism to selectively deliver a real-money gambling ("Teenpatti RM") game. This malicious behavior is specifically triggered for users who install the application via targeted advertising campaigns, with a particular focus on installs originating from YouTube ads. This threat was first discovered in February 2026.

We will demonstrate how the malware leverages a legitimate marketing SDK (`AppsFlyer`) to read the Google Play Store's install referrer data. This data acts as the initial trigger, initiating a chain of events that includes a "persistence bridge" using `PlayerPrefs` to obfuscate the logic flow, communication with a dynamically constructed Command and Control (C2) server for validation, and finally, the delivery of the malicious `AssetBundle` payload over a stealthy **WebSocket connection**.

This analysis provides a complete, evidence-backed walkthrough of this advanced cloaking technique, including the specific analytical steps required to connect each stage, a final Frida script for dynamic verification, and a full list of Indicators of Compromise (IOCs).


## 1. The IL2CPP Challenge & Analysis Methodology

The primary obstacle in analyzing this application is its use of Unity's IL2CPP (Intermediate Language To C++) scripting backend. This ahead-of-time (AOT) compilation process converts the original C#/.NET logic into C++ code, which is then compiled into a native ARM library, `libil2cpp.so`. This effectively removes human-readable MSIL, replacing it with highly optimized and mangled native code, rendering standard .NET Decompilers useless on the final binary.

Our methodology overcame this by adopting a hybrid static-dynamic approach:

1.  **Metadata Reconstruction:** We used **Il2CppDumper** to process `libil2cpp.so` and its corresponding metadata file, `global-metadata.dat`. This step is crucial as it generates a set of analysis aids (`dump.cs`, `script.json`, `stringliteral.json`) that allow us to restore the original C# class and method names to the native disassembly.
2.  **Static Analysis in Ghidra:** We loaded `libil2cpp.so` into Ghidra and applied the scripts generated by Il2CppDumper. This transformed the raw, uncommented disassembly into a navigable and searchable format that mirrors the original code's structure, enabling us to trace the logic.
3.  **Dynamic Verification with Frida:** We used **Frida** and the **`frida-il2cpp-bridge`** to dynamically intercept key functions identified during static analysis, verifying our hypotheses and capturing live data in a controlled environment.

## 2. The Investigation: A Unified Evidence Trail

The cloaking mechanism is not a single check but a chain of carefully decoupled stages designed to thwart analysis. Our investigation followed this chain from the initial trigger to the final payload.

### 2.1. The Initial Clue: The `AndroidManifest.xml`

Like any Android malware analysis, the first step is to inspect the `AndroidManifest.xml` for declared permissions, services, and receivers. This provided the first critical clue:

```xml
<uses-permission android:name="com.google.android.finsky.permission.BIND_GET_INSTALL_REFERRER_SERVICE"/>
```

This permission is the gateway to the **Play Install Referrer API**, allowing the app to query its own installation source. This immediately told us that the application's logic is dependent on how it was installed. The manifest did *not* contain a static `BroadcastReceiver` for the older `com.android.vending.INSTALL_REFERRER` action, confirming the app uses the modern, runtime API.

### 2.2. The First Domino: The Firebase Callback

The presence of the referrer permission, combined with an intent-filter for the `com.appsflyer.referrer.INSTALL_PROVIDER` service in the manifest, strongly indicated that the AppsFlyer SDK was being used to manage the attribution process. The SDK's standard flow involves invoking a callback when attribution data is successfully retrieved.

**Analytical Step:**  To find where this callback was handled, we searched the Il2Cpp dump.cs file for classes implementing AppsFlyer's conversion data listener. This led us directly to the application's custom implementation: `Main.AppsFlyerManager$$onConversionDataSuccess`.

**Evidence & Analysis**
This function contains the first stage of the cloaking logic. Instead of immediately acting on the attribution data, it performs a subtle but critical check.


> **Attacker's Rationale:** By checking for a generic key like `media_source` instead of a specific value (e.g., "youtube"), the developers make the initial trigger broader and less suspicious. The real filtering happens later, on the C2 server.

### 2.3. The Middleman: C2 Validation

With the attribution data now persisted in `PlayerPrefs`, a different part of the application is responsible for the next step: **validating the device with a Command and Control (C2) server.** This stage involves two key functions: one to build and send a detailed device profile, and another to handle the server's response.

#### 2.3.1. Building and Sending the Device Profile

The task of gathering the device profile and sending it to the C2 server is handled by a large, central function, which Ghidra names UndefinedFunction_0102b1c0.

**Analytical Step:** By analyzing the cross-references to functions that read the "AppsflyerData" key, we located this function. Its code reveals a systematic process of fingerprinting the device.

**Evidence & Analysis** This function constructs a detailed profile of the device and the installation, which includes:
- The application version.
- The unique device identifier, generated by Main.GameUtils$$GenDeviceUniqueIdentifier.
- The AppsFlyer ID.
- Crucially, the full contents of the AppsFlyer conversion data, including the media_source.

This profile is then serialized to a JSON string, obfuscated via Main.**`GameUtils$$EncodeRequest`**, and sent as an HTTP POST request to the C2 server at `hxxps://trains[.]circlelanes[.]com`

**Analytical Step:** We examined the different states of this coroutine. After the web request completes (state 2), it decodes the server's JSON response and looks for a specific status code to determine whether to activate the malicious behavior.

### 2.4. The Final Payload: Delivery via WebSocket

The most critical and evasive stage of the attack is the delivery of the final payload. After a successful C2 validation, the application does not download a simple file. Instead, it switches protocols and connects to a different server entirely using a WebSocket to stream the malicious AssetBundle directly into memory.

#### 2.4.1. The WebSocket URL from the C2 Response
The URL for the final WebSocket server is not constructed on the device. Instead, it is delivered directly within the JSON payload of the C2 server's response. This logic is handled within `UndefinedFunction_0102baf8` (part of the `Main.Game.EJMKOCDCFNH$$MoveNext` coroutine).

**Analytical Step:** By analyzing the C2 response handler, we observed that after parsing the JSON, it extracts a specific key containing the final URL, which is then used to initiate the connection.

**Evidence & Analysis**
The function extracts the WebSocket URL from the C2 response before proceeding to the connection phase.

## 3. Dynamic Verification with Frida
To confirm our static findings, we developed a Frida script using the `frida-il2cpp-bridge`. This script allows for real-time interception and manipulation of the application's logic.


**Script Objectives:**

1. **Force Malicious Path:** Intercept the `onConversionDataSuccess` callback and inject a fake "Non-organic" payload to trigger the cloaking logic on demand.
2. **Trace Persistence Bridge:** Hook `PlayerPrefs.SetString` to log the writing of `AppsflyerData` and `enviorment.info`, confirming the data flow between stages.
3. **Capture C2 Payloads:** Hook `Main.GameUtils$$EncodeRequest` and `Main.GameUtils$$DecodeRequest` to capture the plaintext JSON payloads for both the C2 request and response.
4. **Confirm Final Payload:** Hook the `BestHTTP.WebSocket.WebSocket` constructor to log the final WebSocket URL and hook `AssetBundle.LoadFromMemoryAsync` to confirm that a byte array is being loaded into memory, logging its size.

## 4. Conclusion
This Android application demonstrates a sophisticated, multi-stage cloaking technique. It leverages a legitimate marketing SDK (AppsFlyer) to identify users from ad campaigns. It then uses PlayerPrefs as a persistence bridge to decouple its logic, validates the device with a C2 server using an obfuscated, fingerprinted request, and finally uses the C2 response to set a switch (`enviorment.info`) that triggers the use of a persistent WebSocket connection to stream its malicious payload. This multi-stage approach, combined with the use of IL2CPP, effectively hides the malicious logic from static analysis and the final payload delivery from standard HTTP monitoring tools. The successful combination of deep static analysis of the IL2CPP binary and targeted dynamic instrumentation with Frida was essential to uncovering and verifying the complete attack chain, which was first identified in the wild in February 2026.

## Appendix A: Tooling and Replication Workflow

This analysis relies on two primary open-source tools for static analysis of the IL2CPP binary: Il2CppDumper and Ghidra. For readers who are new to Unity game reversing, this section provides a high-level workflow for setting up the analysis environment.

**Prerequisites**
- An extracted APK of the target application (`com.circlelanes.trains`).
- The primary native library, typically found at `lib/arm64-v8a/libil2cpp.so` within the APK.
- The core metadata file, found at `assets/bin/Data/Managed/Metadata/global-metadata.dat`.

**Step 1: Using Il2CppDumper to Generate Analysis Scripts**
Il2CppDumper is a tool that parses the `libil2cpp.so` library and the `global-metadata.dat` file to extract type information and generate scripts that can reconstruct the original code structure.

1. **Download Il2CppDumper:** Obtain the latest release from the official Il2CppDumper GitHub repository.
2. **Execute the Tool:** Run Il2CppDumper and provide it with the paths to the `libil2cpp.so` file and the `global-metadata.dat` file.
3. **Collect the Output:** The tool will generate several files in an output directory. The most important files for our Ghidra analysis are:
    - `script.json`: Contains the addresses and names of all the functions.
    - `stringliteral.json`: Contains the addresses of all string literals.
    - `dump.cs`: A C#-like header file that is useful for reference.

**Step 2: Setting up the Project in Ghidra**
Ghidra is a powerful software reverse engineering suite developed by the NSA.
1. **Download Ghidra:** Obtain the latest version from the official Ghidra website.
2. **Create a New Project:** Start Ghidra, create a new non-shared project, and import the `libil2cpp.so` file.
3. **Analyze the Binary:** When prompted, analyze the file with the default settings. This will perform the initial disassembly.

**Step 3: Applying the Il2CppDumper Scripts in Ghidra**
This is the most critical step, as it applies the reconstructed metadata to the raw disassembly, making it human-readable.

1. **Open the Script Manager:** In your Ghidra project, navigate to `Window` -> `Script Manager`.
2. **Import the Ghidra Script:** Il2CppDumper comes with a Ghidra script (`ghidra_with_struct.py`). Add this script's directory to the Script Manager's settings so Ghidra can find it.
3. **Run the Script:** Find the `ghidra_with_struct.py` script in the manager and run it.
4. **Provide the `script.json` File:** The script will prompt you to select the `script.json` file that you generated in Step 1.
5. **Wait for Completion:** The script will run for several minutes. When it is finished, it will have renamed all the functions and strings in the Ghidra project according to the Il2CppDumper output.

After completing these steps, your Ghidra project will be fully set up for analysis, allowing you to search for class and method names (e.g., `Main.AppsFlyerManager$$onConversionDataSuccess`) and trace the application's logic as we have done in this report.
